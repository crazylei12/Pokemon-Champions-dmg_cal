# Pokemon Champions 对局状态与用户修正流程设计

日期：2026-07-22

## 1. 模块目标

对局状态模块位于图像识别和伤害计算之间，负责保存“当前这局对战里我们已经知道什么、还不知道什么、用户确认过什么”。

它要解决的问题是：

- 图像识别结果不一定可靠，需要用户确认。
- 伤害计算需要明确参数，不能直接吃模糊结果。
- 对战中状态会频繁变化，例如换人、天气、场地、能力等级。
- 用户需要很快修正关键条件，而不是每次打开完整编辑页。

## 2. 总体流程

```text
导入或选择我方保存队伍
  -> 新建对局
  -> 队伍预览截图识别
  -> 用户确认对方队伍，并校验我方保存队伍是否匹配
  -> 对战中截图识别场上宝可梦
  -> 默认填入当前场上攻击方和防守方
  -> 用户选择我方输出或我方承伤
  -> 用户可从双方开局阵容中改选攻击方和防守方
  -> 我方承伤时，用户从数据库合法招式中选择对方招式
  -> 用户快速修正战场条件
  -> 伤害计算
  -> 对局状态继续更新
```

如果用户还没有导入我方队伍，也可以先手动选择我方宝可梦和招式，但伤害计算会缺少完整配置。队伍预览可以跳过，对方也可以手动选择，但识别准确率会下降。

## 3. 对局状态数据

建议维护一个 `BattleSession`：

```text
BattleSession
  sessionId
  createdAt
  gameLanguage
  battleType: SINGLE | DOUBLE | UNKNOWN
  selectedOwnTeamId
  ownTeam: BattlePokemonSlot[6]
  opponentTeam: BattlePokemonSlot[6]
  activeOwnSlots: ActiveSlot[]
  activeOpponentSlots: ActiveSlot[]
  calculationSelection: CalculationSelection
  battleCondition: BattleCondition
  recognitionHistory: RecognitionRecord[]
  userCorrections: UserCorrection[]
```

### 3.1 队伍槽位

```text
BattlePokemonSlot
  slotIndex
  speciesId
  formId optional
  confirmedByUser: boolean
  confidence
  candidates
  linkedBuildId optional
```

我方槽位应优先绑定用户保存的完整配置；对方槽位通常只有种类和模板。

建议另行维护可复用的 `SavedOwnTeam`：

```text
SavedOwnTeam
  savedTeamId
  teamSlotName
  gameTeamId optional
  pokemon: KnownPokemonBuild[6]
  importSource: SCREENSHOT | MANUAL | MIXED
  importStatus: COMPLETE | INCOMPLETE
  userConfirmed: boolean
  createdAt
  updatedAt
```

`SavedOwnTeam` 来自队伍配置界面的两类截图：招式道具页和加点能力页。进入对局时，`BattleSession.selectedOwnTeamId` 指向当前使用的保存队伍。

### 3.2 场上槽位

```text
ActiveSlot
  side: OWN | OPPONENT
  position: SINGLE | LEFT | RIGHT
  speciesId
  sourceTeamSlotIndex optional
  confirmedByUser: boolean
  currentHpPercent optional
  status optional
```

### 3.3 计算选择

`ActiveSlot` 表示当前画面识别到的真实场上宝可梦，`CalculationSelection` 表示用户正在计算的攻击方和防守方。两者不能混为一谈。

```text
CalculationSelection
  direction: OWN_TO_OPPONENT | OPPONENT_TO_OWN
  attackerSide: OWN | OPPONENT
  attackerTeamSlotIndex
  attackerActivePosition optional
  defenderSide: OWN | OPPONENT
  defenderTeamSlotIndex
  defenderActivePosition optional
  defenderAssumption: CURRENT_ACTIVE | SWITCH_IN_FROM_TEAM | MANUAL
  selectedOpponentMoveId optional
  selectedOpponentMoveSource: LEGAL_MOVE_POOL | PRESET | OBSERVED | MANUAL_OVERRIDE optional
  lockedByUser: boolean
```

打开计算界面时，App 用 `activeOwnSlots` 和 `activeOpponentSlots` 初始化 `CalculationSelection`，默认方向是 `OWN_TO_OPPONENT`。用户切到 `OPPONENT_TO_OWN` 后，攻击方变成对方槽位，防守方变成我方槽位，并要求选择一个对方招式。用户一旦改选，`lockedByUser` 变为 `true`，后续截图识别只能提示“场上宝可梦已变化”，不能静默覆盖用户正在看的计算对象。

### 3.4 对战条件

```text
BattleCondition
  weather
  terrain
  ownReflect
  ownLightScreen
  ownAuroraVeil
  opponentReflect
  opponentLightScreen
  opponentAuroraVeil
  statStagesByActiveSlot
  typeOverrideByActiveSlot
  statusByActiveSlot
  customFlags
```

### 3.5 速度线状态

速度线是本局状态的一部分，但不应把对方某个伤害预设当作速度真值：

```text
SpeedLineState
  ownTailwind
  opponentTailwind
  trickRoom
  ownPokemon[teamSlotIndex]: SpeedPokemonModifiers
  opponentPokemon[teamSlotIndex]: SpeedPokemonModifiers

SpeedPokemonModifiers
  stage: -6 .. +6
  paralyzed
  doubled
  choiceScarf optional
```

我方基础速度来自保存队伍的实际能力值，因此在速度线上是一个点。对方基础速度按当前形态的种族速度、`0～32` 速度点以及减速 / 加速性格计算成范围。顺风按队伍分别生效，戏法空间只改变同一先制度中的速度比较方向。

招式先制度来自固定版本的 Champions 招式资源。普通行动始终位于先制度 `0`；已知我方配招和对方可能可学招式里的非守住类先制招式可以生成额外行动标记。守住、看穿、广域防守等保护类先制招式不进入速度线，避免高先制度的保护按钮淹没真正需要比较的行动；帮助、看我嘛等其他先制变化招式仍按实际先制度显示。

## 4. 用户修正原则

### 4.1 识别结果必须可改

任何自动识别结果旁边都应该有修改入口。用户一旦修正，当前对局内应优先使用用户结果。

优先级：

1. 用户手动锁定。
2. 用户确认过的我方保存队伍。
3. 用户确认过的队伍预览结果。
4. 当前截图识别结果。
5. 全图鉴或默认候选。

### 4.2 未知信息必须标出来

不要把未知配置藏起来。界面应该明确显示：

- 对方物种来自头像识别还是用户手动选择。
- 对方努力值未知。
- 对方道具未知。
- 对方特性按默认假设。
- 当前天气由用户手动设置。
- 当前场地未识别。

### 4.3 高频项一键修改

实战中用户没有时间填复杂表单。高频项必须是按钮、开关或小步进器。

高频项包括：

- 计算攻击方，默认当前场上，但可从我方阵容切换。
- 计算防守方，默认当前场上，但可从对方阵容切换。
- 对方耐久预设按队伍槽位分别保存；切换宝可梦再切回来时恢复该宝可梦上次选择的预设，只有物种或形态改变时才清除对应旧预设。
- 计算方向：我方输出 / 我方承伤。
- 我方承伤时的对方招式。
- 天气。
- 场地。
- 墙。
- 能力等级。
- 太晶/类型变化。
- 烧伤。
- 对方防守模板。

## 5. 页面和交互

### 5.1 新建对局页

用户选择：

- 游戏语言。
- 单打或双打，允许自动识别。
- 我方队伍。
- 是否立即开始队伍预览识别。

### 5.2 队伍确认页

展示双方队伍识别结果：

```text
我方队伍                 对方队伍
1. 皮卡丘    已绑定配置   1. 暴鲤龙    94% 图标
2. 风速狗    已绑定配置   2. 烈咬陆鲨  91% 图标
3. 未确认    [选择]       3. 未确认    [选择]
...

[确认队伍] [重新识别]
```

对方队伍预览卡片没有名称文本，确认页里的对方名字来自头像识别候选或用户手动选择。低置信度时应显示裁剪头像、Top-3 候选和搜索入口，不能只给一个看似确定的名字。

头像候选需要显示外观来源：

- 普通或闪光标记，来自识别结果的 `visualVariant` / `isShiny`。
- 模板来源，例如 Champions sprite、PokeAPI Gen 8 icon、PokeAPI shiny artwork 或用户确认截图。
- 置信度和是否低于确认阈值。

如果同一物种同时出现普通和闪光候选，确认页应把它们作为同一个物种的不同外观展示，用户确认后 `BattleSession.opponentTeam` 记录物种身份，外观标记只作为识别证据和后续模板优化信息，不影响伤害计算。

用户确认后进入对战状态。

### 5.3 对战悬浮面板

悬浮面板应该小而密，只放对战中最常用的东西：

```text
方向: [我方输出] [我方承伤]
攻击方: 我方左  皮卡丘      [从我方阵容切换]
目标:   对方右  暴鲤龙      [从对方阵容切换]

十万伏特    82.4% - 97.1%
伏特替换    54.2% - 64.0%
电光一闪     8.1% - 10.0%

[天气] [场地] [墙] [能力] [模板] [重识别]
```

切换到我方承伤时，悬浮面板应变成对方攻击我方的视角：

```text
方向: [我方输出] [我方承伤]
攻击方: 对方右  暴鲤龙      [从对方阵容切换]
招式:   瀑布                [从暴鲤龙可用招式中选择]
目标:   我方左  皮卡丘      [从我方阵容切换]

伤害: 61.3% - 72.5%
前提: 对方常规攻击模板，我方当前配置

[天气] [场地] [墙] [能力] [模板] [重识别]
```

如果目标不是当前场上宝可梦，而是用户从对方阵容里选择的后排宝可梦，面板要明确标注：

```text
目标: 对方后排  暴鲤龙
前提: 假设对方换入承伤
```

### 5.4 攻击方和目标选择弹层

点击“从阵容切换”后，弹层展示开局确认过的队伍，而不是只展示当前场上的宝可梦：

```text
选择目标
[对方左 当前] 暴鲤龙
[对方右 当前] 烈咬陆鲨
[后排] 风速狗
[后排] 败露球菇
[后排] 快龙
[后排] 洗翠月月熊
```

选择后立即刷新伤害结果。当前场上宝可梦只是默认选项，后排宝可梦是合法计算目标。

### 5.5 敌方招式选择弹层

当 `direction` 是 `OPPONENT_TO_OWN` 时，点击“招式”打开敌方招式选择弹层。弹层数据来自数据模块维护的该宝可梦合法招式池。

```text
选择对方招式
暴鲤龙

[搜索招式]

常见 / 已观察
瀑布
水流尾
地震

全部可用攻击招式
咬碎
冰冻牙
水流尾
...
```

规则：

- 只列出该对方宝可梦在当前规则集和形态下可使用的招式。
- 常见预设、已观察招式和本系攻击招式优先显示。
- 变化招式可以折叠在“全部招式”里，选择后伤害结果显示“无直接伤害”或给出特殊规则提示。
- 如果招式池数据缺失，弹层应提示数据缺失，允许用户临时手动补一个招式，并在计算结果中保留 warning。
- 选择后写入 `selectedOpponentMoveId`，并立即刷新承伤结果。

### 5.6 条件快速弹层

点击“天气”“场地”“墙”“能力”等按钮时，出现小弹层：

```text
天气
[无] [晴] [雨] [沙暴] [雪]
```

能力等级使用 `-6` 到 `+6` 的步进器：

```text
皮卡丘
攻击  -1  [ - ] [ + ]
特攻  +2  [ - ] [ + ]
```

### 5.7 完整编辑页

完整编辑页用于慢速配置：

- 我方宝可梦完整配置。
- 对方模板细节。
- 招式列表。
- 特性和道具。
- 个体值和努力值。

悬浮面板只做快速判断，复杂信息交给完整编辑页。

### 5.8 速度线弹层

主悬浮伤害面板顶部把“速度线”作为与“我方输出 / 我方承伤”同级的一级入口。速度线按先制度从左到右分段；同一段内，正常规则下快者在左，戏法空间下慢者在左：

```text
[先制度 +2]       [先制度 +1]             [普通行动 0]
我·快龙·神速 ●    对·玛力露丽·水流喷射 ━  我·振翼发  ●
                                             对·钢铠鸦  ━━━━━
```

- 蓝色点表示我方确定速度。
- 橙色区域表示对方可能速度范围。
- “逐只调整”区域可以直接切换双方宝可梦的当前形态；形态选择与伤害面板共用同一份本局状态，切换后立即重算速度点 / 范围和先制行动。
- 每个槽位都能快速调整速度等级、麻痹、速度翻倍和讲究围巾。
- 顺风按双方分别设置；戏法空间全局设置。
- 高先制度始终先于低先制度；同先制度才比较速度，速度相同仍视为存在随机先手风险。

## 6. 双打目标选择

双打不能只用“我方 vs 对方”两个概念，必须按槽位处理。但槽位只代表当前场上默认目标，用户仍然可以从双方阵容中改选。

推荐 UI：

```text
        对方左       对方右
我方左  [计算]       [计算]
我方右  [计算]       [计算]

其他目标: [对方阵容]
```

用户点任意格子后，下面显示该攻击方全部招式对该目标的伤害。用户点击“对方阵容”后，可以把目标切换成对方任意后排宝可梦，用于估算换入承伤。

切换到我方承伤方向时，矩阵应从对方攻击我方的视角展示：

```text
        我方左       我方右
对方左  [计算]       [计算]
对方右  [计算]       [计算]

其他攻击方: [对方阵容]
其他目标:   [我方阵容]
```

用户点任意格子后，下面显示“对方招式选择 + 该招式打我方目标的伤害”。不能把对方全部合法招式直接当成已知配招；全招式扫描只能作为高级辅助视图。

如果某个槽位未确认，格子显示 `?`，点击后先进入选择宝可梦。

## 7. 对方模板切换

对方模板应该出现在计算结果旁边，并随计算方向变化：

- 我方输出时，切换对方防守模板。
- 我方承伤时，切换对方攻击模板。

```text
对方假设: 常规耐久  [切换]
```

点击后可切换：

- 默认模板。
- 常规耐久。
- 极限物耐。
- 极限特耐。
- 常规攻击。
- 极限物攻。
- 极限特攻。
- 自定义。

切换模板后立即重新计算。

## 8. 状态变化处理

### 8.1 换人

用户点击“重识别”后，图像识别模块更新场上槽位。

如果识别到的宝可梦在已确认队伍中，直接替换当前槽位；如果不确定，展示候选。

如果用户当前正在计算一个手动选择的后排目标，重识别不能直接覆盖该目标。App 应提示：

```text
场上目标已更新为烈咬陆鲨。是否切回当前场上目标？
[保持暴鲤龙换入假设] [切回场上目标]
```

### 8.2 天气和场地

第一版可以完全手动切换。后续如果识别模块能识别图标，再自动给出提示，但用户确认优先。

### 8.3 能力等级

第一版手动维护。用户需要能快速重置：

- 重置当前宝可梦能力等级。
- 重置双方全部能力等级。
- 换人时询问是否清空该槽位能力等级，默认清空。

### 8.4 HP 百分比

HP 百分比不是计算“满血伤害范围”的必要条件，但对 KO 判断很重要。

第一版可以：

- 允许用户手动输入对方当前 HP 百分比。
- 允许用户手动输入我方当前 HP 百分比，用于承伤模式下判断是否会被击杀。
- 如果识别模块能读 HP 条，再作为候选提示。
- 计算输出中显示“当前 HP 下是否可击杀”。

## 9. 数据保存

### 9.1 长期保存

长期保存：

- 用户自己的队伍配置。
- 常用对方模板。
- App 设置。
- 语言和识别偏好。

### 9.2 单局保存

单局保存：

- 本局双方队伍。
- 用户修正记录。
- 当前场上状态。
- 最近计算目标。
- 最近选择的对方招式和承伤方向。

单局数据可以在用户结束对局后删除或归档。

### 9.3 截图保存

默认不长期保存截图。

如果需要保存样本用于调试，应让用户明确开启，并说明：

- 截图只保存在本机。
- 用户可以一键删除。
- 后续如果要导出样本，需要用户主动操作。

## 10. 与其他模块的边界

### 10.1 与图像识别模块

图像识别模块给出候选、来源、置信度和外观标记。队伍预览中的对方槽位通常来自头像识别而不是 OCR；对局状态模块决定是否采纳、是否要求用户确认，并把用户确认后的实体作为本局 `opponentTeam` 的优先来源。`visualVariant` / `isShiny` 用于解释识别依据和积累模板，不参与伤害公式。

### 10.2 与伤害计算模块

伤害计算模块只接受明确的 `DamageRequest`。如果攻击方、防守方、计算方向、对方招式或关键配置未确认，对局状态模块应先让用户选择。

### 10.3 与数据模块

数据模块提供宝可梦、招式、特性、道具、性格等静态数据，并提供“某宝可梦在当前规则集下可使用哪些招式”的合法招式池。对局状态只保存引用 ID 和用户覆盖值。

## 11. MVP 验收标准

第一版对局状态和用户修正流程完成以下能力即可：

1. 用户能创建一局对战。
2. 用户能确认双方队伍，包括通过头像识别或手动选择得到的对方队伍。
3. 用户能选择当前场上宝可梦。
4. 打开计算界面时，当前场上宝可梦自动作为默认攻击方和防守方。
5. 用户能在我方输出和我方承伤两个方向之间切换。
6. 用户能从我方开局阵容中改选我方宝可梦。
7. 用户能从对方开局阵容中改选对方宝可梦，包括后排换入目标。
8. 我方承伤时，用户能从数据库中该对方宝可梦可使用的招式里选择对方招式。
9. 用户能快速修改天气、场地、墙、能力等级、烧伤、对方模板。
10. 修改后伤害结果能立即刷新。
11. 用户修正过的识别结果在本局内保持优先。
12. 用户能查看双方 12 只宝可梦的速度点 / 范围、非守住类先制招式额外行动点，并按槽位修正顺风、戏法空间与速度状态。

## 12. 当前建议

对局状态模块是这个 App 好不好用的关键。识别不可能永远准确，计算也不可能知道对方隐藏配置，所以用户修正必须做得非常快、非常明确。第一版宁可少识别一些，也要让用户一眼知道当前结果基于什么假设。
