# Android 我方队伍 OCR 问题根因与修复说明（2026-07-16）

## 1. 文档目的

本文专门说明手机实测中“我方队伍 OCR”出现的漏项、跨页冲突和误识别问题，包括：

- 最初发现了什么问题；
- 如何排除原图和目录数据问题；
- 根因落在哪些 Android 识别逻辑；
- 实际修改了哪些代码；
- 如何证明修复有效。

完整的逐队现场记录见 [Android 手机我方队伍 OCR 导入记录](android_phone_own_team_ocr_import_record_2026-07-16_zh.md)。本文只解释根因和修复实现。

## 2. 最初发现的问题

在 RMX3820 上用相册中的双页队伍截图导入时，4 组问题样本共出现 14 个异常项：

| 问题类别 | 数量 | 典型现象 | 涉及样本 |
| --- | ---: | --- | --- |
| 能力值漏项 | 4 | 烈咬陆鲨生命或特攻为空 | `Debug05-06`、`Debug07-08`、`Debug21-22` |
| 能力值误识别 | 6 | `182 → 183`、`100 → 40`、雷丘 `99 → 96` | 同上及 `Debug05-06` 雷丘 |
| 两页物种冲突 | 3 | 同一槽位的两页被识别为不同宝可梦 | 3 组样本中的大嘴鸥 |
| 招式漏项 | 1 | 大钢蛇只识别出 3 招，缺少“重磅冲撞” | `Debug21-22` |

另外 4 组样本 `Debug01-02`、`Debug03-04`、`Debug09-10`、`Debug23-24` 没有需要人工补项的问题。

这些错误有两个危险特征：

1. 空字段会进入人工修正流程，比较容易被发现；
2. `183`、`40`、`96` 等错误值本身仍在合法数值范围内，不一定会被“待处理数量”发现，必须对照原图才能识别。

## 3. 排查结论

同一批 4 组问题截图共 8 张在电脑端回放得到 `168/168`，名称和招式目录也都能命中。由此排除了以下方向：

- 原始截图模糊或内容缺失；
- 宝可梦、特性、道具或招式目录缺数据；
- 双页截图本身不对应。

问题集中在 Android 端从“卡片定位”到“候选落盘”之间的处理链路，而且是多处逻辑叠加，不是一个 OCR 模型错误。

## 4. 根因与修复

### 4.1 卡片上沿被模型图拉偏，能力值裁剪整体错位

#### 根因

卡片检测通过紫色区域连通域取得矩形框。烈咬陆鲨等宝可梦模型图与紫色面板连在一起时，模型图也会进入连通域，使右侧卡片的上沿比同一行左侧卡片更高。

第 21 张问题图中，同一行左卡为 `top=336, height=222`，右侧烈咬陆鲨为 `top=311, height=246`。两卡底边基本相同，但右卡上沿向上偏了约 25 像素。能力值格子继续按这个错误高度计算后会整体上移，造成：

- 生命、特攻格没有覆盖完整数字，形成漏项；
- 相邻文字或不完整数字进入裁剪区，形成 `100 → 40` 等误识别；
- 后续多种裁剪结果基于同一个错误卡框，无法靠简单重试恢复。

#### 修复

新增 `normalizeTeamCardRows`：将 6 张卡按行分成 3 组，每行左右两张卡使用共同的紫色面板上、下边界，再进行能力值格子裁剪。

这样保留每张卡自己的左右边界，同时消除模型图只拉高其中一张卡所造成的纵向偏移。

### 4.2 能力值候选按“最大值”落盘，错误结果反而胜出

#### 根因

旧流程会对不同缩放和二值化结果产生多个数字候选，但合并时偏向最大值。因此 `182` 和 `183` 同时出现时会稳定选成 `183`。旧流程还包含单字符乘十、`11 → 111` 一类猜值规则，可能把不完整结果扩展成看似合法的值。

#### 修复

能力值识别改为独立数字链路：

- 在 Gradle 中加入 ML Kit Latin text recognizer，并同步加入对应许可证材料；
- `OwnTeamOcrEngine` 新增 `statRecognizer`，能力值不再完全依赖中文全文识别器；
- `recognizeStatCrop` 对精确裁剪和加宽裁剪分别执行平滑放大与二值化识别；
- `selectStatValueCandidates` 按候选一致性投票，优先采用重复出现的结果；票数相同时先优先保留大于 `32` 的完整值，再按候选出现顺序稳定处理，不再简单取最大值；
- 删除对单字符擅自补零、扩位的猜值路径；
- 最后再通过 `isPlausibleStatValue` 检查合法范围。

这使 `182 / 183 / 182` 收敛到 `182`，`100 / 40 / 100` 收敛到 `100`。

### 4.3 残缺中文名按目录顺序消歧，制造跨页物种冲突

#### 根因

`大嘴` 这类残缺文本可以同时匹配“大嘴鸥”和“大嘴娃”。旧逻辑在两个候选同分时仍会按目录顺序选出一个，导致招式页与能力值页对同一槽位得到不同物种，最终触发两页冲突。

目录中还存在正式拼写和兼容拼写指向同一实体的情况，例如 `Lightning Rod` 与 `Ligtning Rod` 都映射到“避雷针”。如果只按文本候选数量判断，也可能把正确的精确命中误当成歧义。

#### 修复

新增 `selectUnambiguousRecognitionEntity`，先按 `canonicalId` 合并同一实体的别名，再比较不同实体：

- 完整本地化名称的精确命中仍可直接采用；
- 同一实体的正式拼写和兼容拼写不会互相制造冲突；
- 不同实体的低置信度候选在分差不超过阈值时返回未决，不再用目录顺序强行选择；
- `recognizeSpecies`、字段裁剪和 `EntityCatalog.resolve` 统一走这套消歧规则。

结果是残缺的 `大嘴` 不再被武断地落成错误物种，而完整的“大嘴鸥”仍能稳定命中。

### 4.4 第一招裁剪包含属性图标，污染招式文字

#### 根因

大钢蛇第一招的常规裁剪区域包含左侧属性图标，中文 OCR 原始结果一度为 `0重磅冲中撞`，无法与招式目录中的“重磅冲撞”匹配，因此最终只保留另外 3 招。

#### 修复

`entityCropRegions` 为每个招式提供两档区域：

1. 原有的较宽区域；
2. 左边界右移、排除属性图标的纯文字区域。

`recognizeEntityCrop` 先识别常规区域；当结果为空或置信度不足时，再识别纯文字区域，并通过统一消歧逻辑选择结果。真机重试后可得到完整的“重磅冲撞”。

### 4.5 雷丘 `99 → 96`：`6/9` 形态校验只检查了十位

#### 根因

第一轮修复后，第 05 张中的雷丘特攻仍被 ML Kit 稳定识别为 `96`，原图实际为 `99`。当时的形态纠错只在两位数的十位为 `6` 或 `9` 时检查孔洞位置，个位数字从未进入校验。

该个位数字的孔洞纵向中心比例约为 `0.292`，孔洞位于上半部，符合 `9` 而不是 `6`。

#### 修复

第二轮补修将能力值裁剪中的数字连通域从左到右排序，并逐位执行形态校验：

- `statDigitHoleYs` 提取每个可见数字的孔洞位置；
- `digitHoleY` 区分与外部背景连通的空白和数字内部孔洞；
- `correctSixNineDigitConfusions` 对个位、十位和三位数中的每个 `6/9` 都分别校验；
- 孔洞明显位于上部时修正为 `9`，明显位于下部时修正为 `6`，中间模糊区域保持 OCR 原值，避免过度纠错。

因此 `96 + [0.289, 0.292]` 会稳定修正为 `99`。

三位数不会退回旧的“只看十位”逻辑。`correctSixNineDigitConfusions` 对 OCR 数值字符串使用 `mapIndexed`，个位、十位、百位中的每个 `6/9` 都会读取同位置的孔洞特征。例如 `196 + [null, 0.289, 0.292]` 会修正为 `199`；正常的 `182` 在检测到 3 个可见数字时保持 `182`。没有明确上孔或下孔证据的数字保持 OCR 原值，不会为了凑合法值而强制翻转。

### 4.6 整卡 OCR 兜底绕过位数和形态校验

#### 根因

精确裁剪没有返回可用文本时，流程还会采用整卡 OCR 中落在格子附近的数字。第一轮修复后真机回放暴露出奇麒麟特防画面中的 `90—0` 被整卡 OCR 拼成 `900`。该兜底值当时没有经过精确裁剪链路中的数字位数和 `6/9` 校验。

这一问题在真机上经历了两个可区分的状态：最初导出的结果中奇麒麟特防直接成为错误值 `900`；加入合法范围限制后，`900` 被丢弃，于是同一能力值页第一次从空白状态复跑出现 `41/42`。导出草稿逐字段核对后确认，42 个字段中唯一缺少的就是奇麒麟 `actualStats.spd`，其余 41 项均已存在。因此 `41/42` 不是雷丘再次失败，也不是随机丢了一只宝可梦，而是同一个 `90—0 → 900` 兜底误读被拦截后留下的空字段。

#### 修复

`recognizeStatValue` 不再直接接受整卡候选，而是调用 `correctStatValueFromSource`，用同一能力值格子的真实图像重新校正：

- `normalizeStatValueDigitCount` 用可见数字连通域数量约束 OCR 字符位数；画面只有两个数字时，`900` 收敛为 `90`；
- 再执行逐位 `6/9` 形态校验；
- 校正后仍不在合法范围内的候选才被丢弃；
- 最后与精确裁剪候选一起进入一致性投票。

这样精确裁剪和整卡兜底不再使用两套不同的质量门槛。

核心数据流由：

`单个连通域卡框 → 中文整卡 OCR → 猜值/取最大值 → 直接落盘`

调整为：

`同排卡片边界归一化 → 精确/加宽数字裁剪 → Latin OCR 多预处理候选 → 图像形态校验 → 候选一致性投票 → 合法范围检查 → 落盘`

名称与招式链路则增加了：

`常规区域 → 低置信度时纯文字区域重试 → 按实体合并别名 → 歧义时保持未决`

`OwnTeamRecognitionTest` 覆盖以下关键行为：

- `statCandidateSelectionUsesAgreementInsteadOfLargestValue`：证明候选按一致性而不是最大值选择；
- `ambiguousPartialSpeciesNameIsNotResolvedByCatalogOrder`：证明残缺物种名不会按目录顺序强行消歧；
- `moveCropHasATextOnlyRetryThatExcludesTheTypeIcon`：证明招式存在排除属性图标的第二裁剪区；
- `statCropRetriesWithTwoNumberOnlyWidths`：证明左右能力值格都使用精确和加宽两档裁剪；
- `statDigitGeometryCorrectsSixNineConfusionsInEveryPosition`：覆盖个位、十位和三位数中的 `6/9`；
- `statDigitGeometryDropsOcrDigitsThatAreNotPresentInTheCrop`：覆盖 `900 + 2 个可见数字 → 90`。

其中与本轮补修直接对应的数值断言包括：

- `96 + [0.289, 0.292] → 99`：覆盖雷丘个位 `6/9`；
- `196 + [null, 0.289, 0.292] → 199`：覆盖三位数中后两位的逐位纠错；
- `182 + 3 个可见数字 → 182`：证明正常三位数不会被截断；
- `900 + 2 个可见数字 → 90`：覆盖整卡 OCR 多拼出尾随数字的情况。

## 8. 结论

本次问题的根本原因不是截图质量，也不是识别目录缺项，而是 Android 端在边界定位、候选合并、歧义处理和兜底校验上缺少一致的约束。修复后的关键原则是：

1. 先让裁剪区域可靠，再讨论 OCR 结果；
2. 多候选按一致性选择，不用数值大小代替置信度；
3. 歧义结果保持未决，不按目录顺序猜测；
4. 精确裁剪和整卡兜底必须经过相同的图像校验；
5. 合法范围只能拦截明显异常，不能替代逐位视觉校验。

这六项根因均已有对应代码改动、回归测试和真机复测证据。
