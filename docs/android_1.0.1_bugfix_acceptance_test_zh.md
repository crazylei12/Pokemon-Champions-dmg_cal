# Android 1.0.1 问题修复总表与验收测试清单

日期：2026-07-15（Asia/Shanghai）

适用版本：`1.0.1 (4)`

实现提交：`c5cc0f2 fix: harden Android battle workflow and release assets`

## 1. 文档目的

本文汇总本轮代码审查从开始到结束发现的全部高影响问题，包括审查途中追加发现的问题，并给出修复后的最终行为和可直接执行的验收步骤。

测试时应优先完成标记为 P0 的项目。任一 P0 项出现崩溃、旧对局状态污染、核心识别资源缺失、数据恢复破坏原数据或正式签名不一致，都应阻断发布。

### 2.2 安装前注意

`1.0.1` 在尚无正式用户的前提下更换为新的独立生产签名。使用旧签名安装的 `1.0.0` 不能直接覆盖升级到本版本。

- 若设备上是旧签名的 `1.0.0`，先导出需要保留的数据，再卸载旧版并安装 `1.0.1`。
- 从 `1.0.1` 开始，后续正式版本必须持续使用本节记录的生产证书。
- arm64-v8a APK 只用于真实 ARM64 设备；x86_64 APK 只用于对应模拟器。

## 5. 详细验收步骤

### T01（P0）核心特征包随 APK 可用

问题覆盖：BUG-01。

- [ ] 使用干净仓库或当前提交构建 release，不额外复制任何本地模板/cache 文件。
- [ ] 安装与设备 ABI 对应的 APK。
- [ ] 开启飞行模式或断开网络。
- [ ] 授权截图会话，在有效 TEAM_PREVIEW 画面执行双方队伍识别。

预期结果：

- 构建不会提示缺少本机特征文件；
- App 不提示“特征包缺失”“模板格式错误”或哈希不一致；
- 离线状态下仍能给出双方各 6 个槽位的 Top-3 候选；
- 首次可能需要后台读取模板，但不需要联网下载资源；
- APK 内存在 `.bin`、配套 `.json` 和 v2 ROI 三项资源。

失败判定：干净构建依赖开发机未跟踪文件、安装后无法离线识别、APK 内缺任一核心资源。

### T02（P0）特征包篡改和缺失必须阻断发布

问题覆盖：BUG-01、BUG-15。

- [ ] 对正式生成的两个 APK 运行 `npm.cmd run android:verify-release`。
- [ ] 确认两个 APK 均报告 recognition feature pack、production signer 和 ABI 验证通过。
- [ ] 可选：在临时分支中改动特征包一个字节或暂时移走文件，再执行 release 构建；测试后还原临时改动。

预期结果：

- 正常 APK 校验通过；
- 特征包缺失、字节数变化、二进制哈希不匹配或 ROI 哈希不匹配时构建/校验必须失败；
- 不能生成一个“能安装但双方队伍识别必坏”的正式包。

### T03（P0）烧伤和能力等级按宝可梦槽位隔离

问题覆盖：BUG-02。

- [ ] 确认一场对局，选择我方第 1 只，设置烧伤和攻击 `+2`。
- [ ] 切换到我方第 2 只。
- [ ] 检查烧伤和能力等级，再给第 2 只设置防御 `-1`。
- [ ] 切回我方第 1 只。
- [ ] 对对方第 1、2 只重复相同步骤。
- [ ] 收起并重新打开实战面板，再次检查。

预期结果：

- 第 2 只初次打开时为未烧伤、能力等级全 0；
- 切回第 1 只后仍为烧伤、攻击 `+2`；
- 第 2 只只保留自己的防御 `-1`；
- 我方和对方状态互不污染；
- 面板重开后各槽位状态仍正确。

### T04（P0）每次双方队伍重新识别都必须创建新比赛

问题覆盖：BUG-13。这是本轮最重要的业务语义测试。

- [ ] 完成一次双方队伍识别并确认比赛。
- [ ] 在旧比赛中尽量设置多种状态：天气、场地、反射壁/光墙/极光幕、Protect、烧伤、能力等级、Helping Hand、暴击、AOE、形态覆盖、对手现场预设/道具、速度线和逐只速度修正。
- [ ] 记住一个明显的旧伤害结果。
- [ ] 不论画面中的双方队伍是否变化，再次执行一次双方队伍识别。
- [ ] 先不要点击“确认并开始对战”，尝试打开旧实战面板。
- [ ] 然后确认新识别结果，进入新比赛。

预期结果：

- 新识别结果保存成功时，旧 `current-battle-session.json` 立即作废；
- 即使第二次识别的是完全相同的 12 只宝可梦，也被视为新比赛；
- 未重新确认前不能继续使用旧面板，应进入本局确认流程；
- 新比赛中的天气、场地、墙、Protect、烧伤、能力等级、形态、手动道具/预设、速度线等全部回到默认值；
- 旧伤害结果和旧招式选择不能直接沿用；
- 如果第二次识别失败或取得不可见空帧，不能覆盖上一次有效结果，也不能误删仍有效的旧比赛。

### T05（P1）同一比赛内切换我方保存队伍

问题覆盖：BUG-12。

- [ ] 在已确认比赛中使用队伍 A。
- [ ] 为队伍 A 设置非默认我方形态、我方第 2 只烧伤/能力等级、我方逐只速度修正，并选择一个招式。
- [ ] 在实战面板切换到队伍 B。
- [ ] 检查我方选择和状态，同时检查对手预设及天气/场地等公共状态。

预期结果：

- 我方自动回到队伍 B 的第 1 只；
- 旧招式选择、队伍 A 的形态覆盖、逐只烧伤/能力等级和逐只速度修正被清空；
- 队伍 B 不继承队伍 A 的逐只状态；
- 因为仍是同一场比赛，天气、场地、墙、对手身份及对手状态可以继续保留；
- 若希望全部清空，应重新进行双方队伍识别，而不是仅切换保存队伍。

### T06（P1）对手现场道具覆盖的三态语义

问题覆盖：BUG-06。

对每种状态应用后执行一次伤害计算，并关闭/重开面板检查持久化：

- [ ] “沿用原预设”：保持预设原有道具。
- [ ] “明确无道具”：清除一个原本带道具的预设。
- [ ] 从当前版本 Pokémon Champions 已实装的道具目录中指定至少一种实际道具，例如“讲究围巾”。
- [ ] 确认尚未在当前游戏版本实装的道具不会被误当作应用目录缺失。
- [ ] 切换到另一只对手，再切回。
- [ ] 点击“恢复原预设”。

预期结果：

- “沿用”与“明确无道具”不会被混为同一状态；
- 指定道具后使用 `MANUAL_CURRENT` 现场假设立即重算；
- 当前游戏版本已实装的道具可以从目录准确选择；尚未实装的道具不作为验收要求；
- 覆盖只作用于本场当前对手槽位，不修改原始预设文件；
- “恢复原预设”后能力点、性格、特性和道具覆盖一起撤销。

### T07（P0）可用于计算的完整性判断

问题覆盖：BUG-07、BUG-11。

分别让一只宝可梦处于以下状态，保存后返回首页，并重新进入对局确认页：

- [ ] 六项实际能力值完整、特性存在、至少 1 个招式；
- [ ] 缺任意一项实际能力值；
- [ ] 缺特性；
- [ ] 0 个招式；
- [ ] 手动把旧 JSON 的 `damageReady` 写成 `true`，但实际仍缺特性或招式（开发者可选测试）。

预期结果：

- 只有“六项实际能力值 + 特性 + 至少一个招式”同时满足时才显示“可用于计算”；
- 缺项时首页“用于计算”按钮不可用，并显示第一条具体缺失原因；
- 不完整队伍不会出现在“确认本局双方队伍”的可选队伍中；
- 已存在对局引用到不完整队伍时，实战面板不会绕过检查；
- App 每次加载都会重新计算 readiness，不信任旧文件中可能错误的 `damageReady=true`。

### T08（P1）1–3 个招式合法，空技能槽只提醒

问题覆盖：BUG-08。

- [ ] 手动编辑一只宝可梦，只保留 1 个招式并保存。
- [ ] 分别测试 2 个、3 个招式。
- [ ] 测试 4 个招式。
- [ ] 清空全部招式。
- [ ] 使用我方双页 OCR，准备一只只识别到 1–3 个真实招式的宝可梦。
- [ ] 在 OCR 手动修正页留空未携带的技能槽。
- [ ] 尝试录入重复招式。

预期结果：

- 1–3 个招式都能保存并用于计算；
- 界面显示“已设置 N/4，空技能槽允许保留”的提醒；
- 4 个招式不显示部分招式提醒；
- 0 个招式不能用于计算；
- OCR 不会仅因为没有识别满 4 招而强制补满；
- OCR 修正仍要求至少一个招式，并阻止重复招式；
- 招式数量不足不会掩盖特性、能力值或宝可梦冲突等真正阻断项。

### T09（P0）手动整包备份与恢复

问题覆盖：BUG-04。

- [ ] 创建至少两支已保存队伍。
- [ ] 创建并确认一场比赛，设置若干对局状态。
- [ ] 如条件允许，保留一份未完成的我方双页 OCR 草稿。
- [ ] 在设置页切换更新频道。
- [ ] 选择“导出队伍与对局备份”，保存 JSON。
- [ ] 删除或修改本机队伍、对局和更新频道。
- [ ] 选择“从备份恢复”，确认“校验并恢复”。
- [ ] 恢复后重启 App，再检查数据。

预期结果：

- 导出提示正确的队伍数量以及是否包含当前对局；
- 备份包含保存队伍、当前预览/对局、未完成导入草稿和更新频道；
- 备份不包含原始截图；
- 恢复前有明确的整包替换确认；
- 恢复后队伍数量、队伍配置、当前对局、草稿和更新频道与备份一致；
- 恢复完成后首页和对局页立即或重启后显示恢复数据。

### T10（P0）损坏备份必须拒绝并保留原数据

问题覆盖：BUG-04。

复制一份有效备份，在副本中分别制造以下错误，每次导入前先在 App 中创建一份可辨认的本机队伍：

- [ ] 把 `kind` 改成其他字符串；
- [ ] 把 `schemaVersion` 改成不支持的版本；
- [ ] 给两个队伍使用相同 ID；
- [ ] 把队伍改成不是 6 只宝可梦；
- [ ] 让当前对局引用一个备份中不存在的我方队伍；
- [ ] 把对方阵容改成不是 6 只；
- [ ] 可选：使用超过 16 MB 的文件。

预期结果：

- 导入显示具体失败信息；
- 校验失败时不会先删除本机数据；
- 写入中途失败时自动恢复导入前快照；
- 测试前创建的本机队伍、原当前对局和更新频道仍保持原样；
- 不能出现只恢复一半队伍的混合状态。

系统备份可选检查：在支持 Android Backup/设备迁移的测试环境中，验证 `saved-teams/`、`battle-session/`、两个导入 JSON 和更新设置会迁移；截图、缓存和识别原图不得进入备份。

### T11（P0）双方队伍识别关闭竞态与迟到回调

问题覆盖：BUG-03、BUG-10。

- [ ] 记录当前双方队伍预览文件和对局状态。
- [ ] 启动截图会话并触发双方队伍识别。
- [ ] 在识别开始后的 0–1 秒内结束悬浮识别会话或撤销截图会话。
- [ ] 等待 15 秒，观察 App 和系统。
- [ ] 重复至少 10 次，包括首次冷加载模板时。
- [ ] 再次正常启动会话并完成一次识别，确认后续功能仍可用。

预期结果：

- 不出现 OpenCV Mat 已释放、模板关闭、线程拒绝等崩溃；
- 结束会话后不再弹出本局确认窗口；
- 迟到结果不写入新的预览或 BattleSession；

## 7. 发布阻断标准

出现以下任一情况，不应发布：

- APK 内缺少核心 `.bin`、元数据或 ROI，或任一哈希不一致；
- 每次重新识别成功后仍能继续使用上一场比赛状态；
- 不同宝可梦槽位互相继承烧伤或能力等级；
- 识别过程中关闭服务会崩溃、弹出幽灵窗口或在结束后写入文件；
- 缺特性或 0 招式的队伍仍可进入计算；
- 1–3 个合法招式被强迫补满；
- 对手道具不能明确设置或清除；
- 损坏备份导入后破坏原数据；
- release 证书不是预期生产证书，或 debug/release 使用同一证书；
- ARM64 真机包或 x86_64 模拟器包包含错误 ABI；
- 任一 P0 测试未通过且没有明确记录的阻断原因。
