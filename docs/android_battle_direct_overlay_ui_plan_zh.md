# Pokemon Champions 对战原位覆盖 HUD 方案

日期：2026-07-22

状态：方案草案，尚未实现

方案分支：`plan/battle-direct-overlay-ui`

设计基线：`main` @ `c7c4cfc`

## 1. 背景

当前实战流程要先点悬浮球、打开伤害面板，再在一整块面板里逐项选择我方宝可梦、对手宝可梦、招式和其它条件。功能完整，但用户在每回合都要把注意力从游戏画面移到辅助面板，操作成本偏高。

本方案新增“对战原位 HUD”：不再把最高频功能集中到一张大面板，而是把它们放到 Pokemon Champions 已有的战斗信息附近。

用户进入对局并确认双方阵容后，应当直接看到：

- 顶部的先制度 `0` 速度数轴。
- 对手右上角名字附近的两个宝可梦选择控件。
- 我方左下角名字附近的两个宝可梦选择控件。
- 选中我方攻击方和对手目标后，位于我方名字上方的四招伤害百分比。

详细配置面板继续保留，负责低频编辑和兜底，不再是查看我方输出伤害的必经入口。

## 2. 设计依据与首批适配范围

本方案检查了本机原始资料目录 `D:\pokemon-champion-dmg_cal\docs\pic\1 (2)\3` 中的 14 张真实对战截图。截图均为 `3392 × 2400` 横屏画面，比例约为 `1.413:1`，布局表现稳定：

- 对手两个名字与 HP 条固定在右上区域。
- 我方两个名字与 HP 条固定在左下区域。
- 战斗、宝可梦等游戏操作按钮固定在右侧。
- 天气文字可能出现在左上，场上模型和特效会覆盖中部，因此中部不适合放常驻大面板。

首批布局只以这组横屏截图和当前 Android 安全区为已知基线。其它分辨率、长宽比和设备必须通过布局配置与实机验收后才能列为支持范围，不能把这 14 张截图当作全设备证明。

## 3. 目标与边界

### 3.1 目标

1. 阵容已确认后，无需再次打开大面板即可看到默认攻防组合的伤害。
2. 切换攻击方或目标时，主要流程控制在一次点选；需要换成后排宝可梦时，通过一次下拉选择完成。
3. 四个招式固定按我方保存配招的第 `1～4` 位显示，只显示序号和伤害比例，不显示招式名。
4. 速度数轴只展示先制度 `0` 的普通行动，避免先制招式标签挤占战斗画面。
5. 除明确的选择控件外，覆盖层不能阻挡游戏触控。
6. 所有选择仍写入同一份本局状态，原伤害面板、速度线和新 HUD 不能各自维护一套互相冲突的数据。

### 3.2 非目标

- 不注入或修改游戏 UI。
- 不自动点击游戏，不读取游戏内存，也不根据画面自动替用户下达操作。
- 第一阶段不做逐帧战场识别；当前是谁仍来自已确认阵容、已有识别结果和用户手动选择。
- 不在原位 HUD 中复制完整的对手配置编辑器、能力等级表或合法招式全集。
- 不删除现有详细面板；它仍是我方承伤、对手招式选择和复杂条件编辑的完整入口。

## 4. 最终画面结构

```text
┌────────────── 先制度 0：快 ← 速度数轴 → 慢 ─────────────────────┐
│ [状态: 雨·我方顺风]                         [对手左⌄] [对手右⌄] │
│                                                                  │
│                         Pokemon Champions 战斗画面                │
│                                                                  │
│ [1 33.6–39.9%] [2 0%] [3 51.2–61.0%] [4 —]                      │
│ [我方左⌄] [我方右⌄]                               游戏操作按钮区 │
└──────────────────────────────────────────────────────────────────┘
```

上图是结构示意，不代表最终像素尺寸。HUD 使用透明或低透明背景，不复制游戏原有 HP 条，也不覆盖右侧游戏操作按钮。

### 4.1 基线锚点

以下坐标是从 `3392 × 2400` 截图总结出的首版起点，均相对于当前可用安全区归一化。实现时应放进布局配置，不能散落成硬编码像素：

| 元素 | 建议横向范围 | 建议纵向范围 | 说明 |
| --- | --- | --- | --- |
| 先制度 0 数轴 | `0.17～0.98` | `0.00～0.035` | 贴顶的被动绘制层，不放大段背景 |
| 对手左选择器 | `0.55～0.76` | `0.02～0.085` | 只覆盖名字带，避开 HP 百分比 |
| 对手右选择器 | `0.76～0.97` | `0.02～0.085` | 下拉向下展开 |
| 我方左选择器 | `0.03～0.23` | `0.865～0.925` | 下拉向上展开 |
| 我方右选择器 | `0.23～0.43` | `0.865～0.925` | 下拉向上展开 |
| 四招伤害条 | 跟随已选我方选择器 | 选择器上方约 `0.055` 高 | 只在已选攻击方上方显示 |

布局计算必须以 `OverlaySafeAreaProvider.currentRegion()` 返回的当前可用区域为基准，继续响应显示区域、系统栏、刘海、折叠区和横竖屏变化；不能重新使用 `maximumWindowMetrics.bounds`。

## 5. 核心交互

### 5.1 显示时机

1. 用户识别并确认双方阵容。
2. App 建立 `BattleSession`，默认选择我方和对手各一个槽位。
3. 原位 HUD 自动出现并立刻计算默认组合，不要求用户再点悬浮球。
4. 悬浮球或边缘把手改为“显示/隐藏 HUD、打开详细面板、重新确认阵容”的入口。

如果没有完整我方队伍、没有本局对手阵容或伤害引擎尚未准备好，HUD 不伪造结果，而是显示短占位状态，并提供进入详细面板的入口。

### 5.2 选择攻击方与目标

每个名字控件拆成两个触控区：

- 名字主体：选中该位置作为攻击方或目标。
- 右侧小箭头：打开该位置的宝可梦下拉列表。

我方名字主体只负责选择我方攻击方，对手名字主体只负责选择对手目标。选择顺序允许任意，但推荐引导语为“先点我方，再点目标”。阵容建立后已有默认组合，因此用户什么都不点也能看到第一组结果。

选中状态使用明显但不刺眼的描边：

- 当前我方攻击方：金色描边。
- 当前对手目标：青色描边。
- 未参与本次计算的其它名字控件：低透明度。

### 5.3 下拉列表

下拉列表始终展示本局确认过的 6 只宝可梦，而不是只展示当前画面上的两只：

```text
选择对手右侧计算对象
1. 暴鲤龙       [当前]
2. 烈咬陆鲨     [当前]
3. 风速狗       [后排]
4. 败露球菇     [后排]
5. 快龙         [后排]
6. 洗翠月月熊   [后排]
```

规则：

- 对手选择器位于屏幕上方，列表向下展开；我方选择器位于屏幕下方，列表向上展开。
- 选择后立即关闭列表、选中该控件并重新计算。
- 选择后排宝可梦时，名字控件显示一个小型“换入”标记，明确这是模拟对象，不是假装识别到的场上状态。
- 有多个合法形态时，在同一行提供二级形态入口，并复用现有 `ownFormOverrides` / `opponentFormOverrides`。
- 列表只覆盖自己的矩形区域；点击列表外部时关闭，不吞掉其余游戏触控。

名字位置只是 HUD 的显示槽位。把后排宝可梦放到“对手右”不应改写队伍识别结果，也不应宣称它真的已经换上场。

### 5.4 四招伤害条

选择我方攻击方和对手目标后，在已选我方名字控件正上方显示：

```text
[1 33.6–39.9%] [2 0%] [3 51.2–61.0%] [4 —]
```

展示规则：

- `1～4` 永远对应保存队伍中的原始配招顺序，不能按拼音、属性或伤害重新排序。
- 默认显示当前对手配置下的最小～最大伤害百分比，保留一位小数。
- 直接伤害恒为 `0` 时显示 `0%`。
- 守住、变化招式或其它无直接伤害结果显示 `—`。
- 招式缺失或计算失败显示 `?`，不能把它渲染成 `0%`。
- 计算进行中显示四个短占位符；旧目标的结果必须立即变暗，不能在新目标名下暂时展示旧伤害。
- 每个位置不显示招式名。用户通过游戏内第几个招式和保存配招顺序对应。

对手模板未知不能被隐藏。为保持伤害条本身只有“序号 + 百分比”，在对手名字控件旁单独显示全局假设标记，例如 `≈ 默认耐久`。点击该标记进入现有对手配置面板。

### 5.5 先制度 0 速度数轴

顶部速度条复用现有速度计算，但只过滤 `priority == 0` 的行动：

- 我方确定速度继续显示为蓝色点。
- 对手可能速度继续显示为橙色范围。
- 默认情况下左边更先行动，即数值从较快到较慢排列，并显示“快 ←”提示。
- 戏法空间开启后，同一先制度内改为慢者在左，并显示醒目的“空间”状态。
- 顺风、速度等级、麻痹、速度翻倍和讲究围巾继续参与有效速度计算。
- 默认绘制双方 12 只宝可梦的点或范围；只给当前两个场上位置和当前计算组合显示短标签，其余只显示 `我1～我6` / `对1～对6` 的弱化标记，避免 12 个全名重叠。
- 当前攻击方和目标的点或范围提高亮度，便于直接判断所选组合的速度关系。
- 非守住类先制招式不在这条原位数轴中显示；它们继续留在完整速度线面板。

速度数轴本身是不可触摸的被动绘制层。旁边只保留一个小型“速度设置”按钮，打开现有完整速度线页面。

### 5.6 状态与详细功能

原位 HUD 显示一组只占一行的当前状态胶囊，例如：

```text
[雨] [电场] [我顺] [对顺] [空间] [墙*]
```

只显示已启用状态；全部默认时显示 `[状态: 默认]`。点击状态胶囊打开现有快速条件面板。这样用户能知道当前百分比为什么变化，但不会把所有能力等级步进器常驻铺在游戏画面上。

现有功能的去向如下：

| 当前功能 | 原位 HUD | 详细入口 |
| --- | --- | --- |
| 选择我方/对手 | 直接覆盖在双方名字附近 | 保留原面板选择器 |
| 我方四招输出 | 名字上方四个序号伤害 | 原面板可看招式名和详细说明 |
| 先制度 0 速度 | 顶部数轴 | 完整速度线保留所有先制度和逐只调整 |
| 天气、场地、顺风、空间、墙 | 状态胶囊显示，点击快速修改 | 完整条件页 |
| 对手模板 | 对手名字旁显示当前假设 | 对手配置页 |
| 形态 | 名字下拉中的二级入口 | 原面板形态选择器 |
| 我方承伤与对手招式 | 第一阶段不常驻 | 原面板完整支持；后续可做反向原位模式 |
| 重新识别/重新确认阵容 | 边缘把手菜单 | 原设置流程 |

## 6. 状态模型

### 6.1 保持一个权威状态源

现有 `BattleCalculationState.ownSlot` 和 `opponentSlot` 继续代表当前正在计算的攻防组合。新 HUD、旧面板和伤害请求都读取这两个值，任何一处选择变化后都写回同一份 `BattleSession`。

需要新增的只是“两个可视位置当前摆了队伍里的谁”，不能把它与权威的队伍顺序混在一起：

```text
BattleDirectHudState
  visible
  ownDisplayAssignments
    LEFT  -> ownTeamSlotIndex
    RIGHT -> ownTeamSlotIndex optional
  opponentDisplayAssignments
    LEFT  -> opponentTeamSlotIndex
    RIGHT -> opponentTeamSlotIndex optional
  selectedOwnPosition
  selectedOpponentPosition
```

双打默认把队伍槽位 `0/1` 放到左右显示位置；单打只启用一个位置。用户从下拉列表换成后排时，只更新显示分配，并在该显示位置被选中时同步更新 `ownSlot` 或 `opponentSlot`。

### 6.2 识别状态与模拟选择分离

后续如果对战场地识别补充了真实 `activeOwnSlots` / `activeOpponentSlots`，它只负责提示当前场上变化。用户正在看的换入假设和手动选择不能被静默覆盖。

## 7. 伤害计算改造

当前 `buildBattleDamageRequest()` 在两个方向都固定生成 `moveSelection.mode = ONE_MOVE`，`parseOverlayResult()` 也只读取第一个结果。原位 HUD 不应连续调用四次引擎，而应一次请求四招：

```text
calculationDirection: OWN_TO_OPPONENT
moveSelection:
  mode: ALL_ATTACKER_MOVES
```

建议新增独立的 `buildDirectHudDamageRequest()` 与 `parseDirectHudDamageResult()`，或者给现有构建器加入明确的请求模式参数。不要用界面当前的拼音/属性排序结果构造 HUD 请求。

解析结果时：

1. 以我方保存配招的 canonical/showdown ID 顺序为主键。
2. 把引擎返回的 `moveResults` 映射回原始 `1～4` 位置。
3. 同时保留 warnings，但只在 HUD 上汇总成一个假设/警告标记。
4. 使用递增 generation 或不可变请求键丢弃过期异步结果。
5. 请求键至少包含双方槽位和形态、对手配置、四个招式、天气/场地/墙、能力等级、异常状态、双打范围修正。

可增加小型内存缓存，使用户在两个已看过的目标之间切换时立即恢复结果；缓存只服务显示，不替代权威会话状态。

## 8. Android 覆盖窗架构

不能创建一个可触控的透明全屏 `TYPE_APPLICATION_OVERLAY` 再指望透明区域把触控自动传给游戏。Android 窗口命中后，即使 View 返回 `false`，也不能把该次触控可靠地重新分发给下层游戏窗口。

推荐拆成三类窗口：

1. **被动绘制窗口**：速度数轴和四招伤害条，使用 `FLAG_NOT_TOUCHABLE | FLAG_NOT_FOCUSABLE`。
2. **小型交互窗口**：四个名字选择器、状态胶囊和边缘把手；窗口尺寸只覆盖实际触控目标，继续使用 `FLAG_NOT_FOCUSABLE | FLAG_NOT_TOUCH_MODAL`。
3. **临时弹层窗口**：宝可梦下拉、形态选择和快速状态面板；仅弹出时占用自己的矩形，外部点击负责关闭。

可在现有 `BattleOverlayController` 旁新增 `BattleDirectHudController`，由一个轻量协调器统一处理以下事件：

- 加载/保存 `BattleSession`。
- 在 HUD 与详细面板之间同步选择。
- 显示模式切换和横竖屏处理。
- 分发安全区变化。
- 取消过期伤害计算。

`BattleDirectHudController` 不应自行复制队伍仓库、对手预设或伤害引擎逻辑，只消费现有 `TeamRepository`、`BattleSessionRepository`、`OpponentPresetRepository`、`DamageEngineRuntime` 和速度线计算函数。

### 8.1 横竖屏与安全区

- 游戏横屏且存在有效对局时显示原位 HUD。
- 切到竖屏时隐藏所有原位窗口但保留选择状态，恢复横屏后重新布局和显示。
- 继续响应 `DisplayManager.DisplayListener` 的即时与延迟刷新，验证“已经打开的 HUD”在旋转后会移动，而不只是下次打开时正确。
- 每种支持比例使用一个 `BattleHudLayoutProfile`；找不到匹配配置时回退为详细面板，不在错误位置显示可点击控件。

### 8.2 样式原则

- 使用与现有辅助 App 一致的深色半透明底、金色/青色选中描边，避免做成与游戏官方 UI 难以区分的仿制样式。
- 可读文字最小 `12sp`，交互目标最小 `44～48dp`。
- 名字过长时单行省略，完整名字在下拉列表显示。
- 四招伤害条允许根据宽度缩成两行 `2 × 2`，但序号顺序始终为左上、右上、左下、右下的 `1～4`。

## 9. 显示状态与错误处理

| 状态 | HUD 表现 |
| --- | --- |
| 引擎加载中 | 四个位置显示短占位符，不显示旧组合结果 |
| 我方配招不足四个 | 缺失位置显示 `—`，实际存在的招式保持原序号 |
| 对手配置未知 | 显示 `≈ 默认耐久`，伤害仍可计算 |
| 对手物种未确认 | 目标控件显示 `?`，不发起伤害请求 |
| 招式无法计算 | 对应位置显示 `?`，全局警告点亮 |
| 无直接伤害 | 对应位置显示 `—` |
| 用户选择后排 | 对应名字显示“换入”标记 |
| 画面比例不受支持 | 隐藏原位 HUD，提示使用详细面板 |
| 进入竖屏 | 暂时隐藏，返回横屏后恢复 |

## 10. 实施阶段

### 阶段 A：状态和四招计算

- 增加 `BattleDirectHudState` 和左右显示位置映射。
- 增加 `ALL_ATTACKER_MOVES` 请求构建与四结果解析。
- 保证原配招顺序、无伤害占位、warning 和过期结果处理正确。
- 为请求构建、结果映射和状态同步补单元测试。

### 阶段 B：名字选择器与伤害条

- 按基线布局配置实现四个小型交互窗口。
- 实现向上/向下的 6 只宝可梦下拉列表和后排标记。
- 实现不可触摸的四招结果窗口。
- 从阵容确认结束后自动进入 HUD，并保留详细面板入口。

### 阶段 C：顶部速度数轴与状态胶囊

- 从现有 `speedLineActions()` 过滤先制度 `0` 并实现紧凑绘制。
- 对当前组合高亮，处理顺风、麻痹、围巾、速度等级和戏法空间。
- 把已启用的天气、场地、墙和速度状态投影为状态胶囊。

### 阶段 D：双分支和实机验收

这是共享战斗 UI，不属于录像专属功能。正式实现时必须分别落到：

- `main` 标准版。
- `feature/battle-replay-phase-4` 录像版。

两条分支保留各自的 `OverlayCaptureService` 架构，手工对齐行为和测试，不整体合并分支。录像版还要验证新增 HUD 不进入游戏录像，继续满足“无辅助浮层、无麦克风、无其它 App 音频”的范围。

## 11. 验收标准

### 11.1 功能

1. 双方阵容确认后，原位 HUD 自动出现，默认组合无需再点悬浮球即可看到四招结果。
2. 点任意我方名字主体后，再点任意对手名字主体，四招伤害在已选我方名字上方刷新。
3. 点击箭头可以从 6 只阵容中选择宝可梦；后排选择有清楚标记。
4. 四招顺序始终与保存配招 `1～4` 一致，不显示招式名。
5. 速度数轴只包含先制度 `0`，并正确应用顺风、麻痹、围巾、速度等级和戏法空间。
6. HUD 修改与详细面板修改双向同步，关闭再打开仍保留本局选择。

### 11.2 触控与布局

1. 除名字选择器、状态胶囊和边缘把手外，覆盖区域全部不拦截游戏触控。
2. 右侧战斗和宝可梦按钮在 HUD 显示时仍可正常点击。
3. 14 张 `3392 × 2400` 基线画面上，选择器与对应名字区域对齐，伤害条不覆盖我方 HP 数值。
4. 已打开 HUD 经过横屏 → 竖屏 → 横屏后能恢复到当前安全区中的正确位置。
5. 其它设备只有在完成对应布局配置和实机验证后才标记为支持。

### 11.3 计算与性能

1. 一次伤害引擎调用返回最多四个我方招式结果，不连续发起四次计算。
2. 快速连续切换目标时，只显示最后一次选择的结果。
3. 引擎已就绪且命中本地数据时，选择变化到四招结果稳定显示的目标延迟不高于 `300 ms`。
4. 状态招式、免疫、配置缺失和引擎错误能与 `0%` 正确区分。

### 11.4 双分支与录像

1. 标准版和录像版分别通过相关单元测试、`assembleDebug` 和 `git diff --check`。
2. 至少在一台手机和一台平板上验证选择、下拉、游戏触控透传和旋转恢复。
3. 录像版实际录制样本中不能出现原位 HUD、下拉框、状态胶囊或详细面板。

## 12. 建议采用的默认决策

- HUD 默认自动显示，不让用户每回合重新开启。
- 四招显示最小～最大百分比，而不是只显示最大值或平均值。
- 四招固定跟随保存配招顺序，不提供 HUD 内排序。
- 速度数轴默认包含双方 12 只的先制度 `0` 标记，但只突出当前场上和当前计算组合。
- 详细面板继续承担我方承伤、对手招式、完整配置和复杂条件；原位 HUD 首先解决最高频的我方四招输出路径。
- 使用多个窄小覆盖窗口，拒绝可触控的透明全屏窗口。

这些默认值能在不牺牲计算前提和游戏可操作性的情况下，最大程度减少实战中“打开悬浮球—展开面板—逐项调整”的步骤。
